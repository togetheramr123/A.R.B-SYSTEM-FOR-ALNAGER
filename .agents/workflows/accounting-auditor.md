# قاعدة المراجعة المحاسبية المستمرة (Accounting Auditor)

## متى تُطبق؟
تُطبق **تلقائياً** بعد كل تعديل يمس أياً من:
- Server Actions (المشتريات، المبيعات، المحاسبة، المخزون)
- دورات العمل (Business Cycles)
- حقول قاعدة البيانات المالية (amounts, quantities, costs)
- مكونات الفواتير والمدفوعات

## ماذا يفعل المراجع؟

### 1. فحص سلامة الدورة المستندية (Document Cycle Check)
بعد كل تعديل، تحقق أن هذه الدورات لا تزال تعمل بشكل متكامل:

**دورة المشتريات:**
```
طلب عرض سعر (Draft) → تأكيد (Purchase) → [إذن استلام مخزني يُنشأ تلقائياً]
→ استلام البضاعة (Validate Picking) → [qtyReceived يتحدث]
→ إنشاء فاتورة (createBillFromOrder) → [الكمية = Received - Invoiced]
→ ترحيل الفاتورة (confirmInvoice) → [AVCO يتحدث + قيد محاسبي]
→ تسجيل الدفع (registerPayment) → [حساب المورد يُسدد]
```

**دورة المبيعات:**
```
عرض سعر (Draft) → تأكيد (Sale) → [إذن تسليم مخزني يُنشأ تلقائياً]
→ تسليم البضاعة (Validate Picking) → [qtyDelivered يتحدث]
→ إنشاء فاتورة (createInvoiceFromOrder) → [الكمية = Delivered - Invoiced]
→ ترحيل الفاتورة (confirmInvoice) → [قيد محاسبي Anglo-Saxon + COGS]
→ تسجيل الدفع (registerPayment) → [حساب العميل يُسدد]
```

**دورة المرتجعات:**
```
إذن مخزني مكتمل → زر إرجاع → [إذن عكسي يُنشأ]
→ اعتماد المرتجع → [المخزون يرجع + الكميات تتحدث]
→ إنشاء إشعار دائن (Credit Note) → [فاتورة عكسية]
→ ترحيل الإشعار → [قيد عكسي]
```

### 2. فحص التوازن المحاسبي (Accounting Balance Check)
بعد أي تعديل على القيود أو الفواتير، تحقق أن:
- **مجموع المدين = مجموع الدائن** في كل قيد يومي (هذا قانون أساسي لا يمكن كسره)
- **amountResidual** محسوب بشكل صحيح (الإجمالي - المدفوع)
- **قيد الفاتورة** يحتوي على الأقل: سطر حساب الشريك + سطر الإيراد/المصروف

### 3. فحص تكامل المخزون (Inventory Integrity Check)
- **quantityOnHand** يتغير فقط عند اعتماد إذن مخزني (Validate Picking)، وليس عند إنشاء الفاتورة
- **costPrice (AVCO)** يتحدث فقط عند ترحيل فاتورة مشتريات
- **qtyReceived / qtyDelivered** يتحدث فقط عند اعتماد الإذن المخزني
- **qtyInvoiced** يتحدث فقط عند إنشاء الفاتورة

### 4. اكتشاف الثغرات والاقتراحات (Gap Analysis)
في نهاية كل جلسة عمل أو عند طلب المستخدم، قدّم تقريراً مختصراً:

```
📊 تقرير المراجعة المحاسبية:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ الدورات السليمة: [قائمة]
⚠️ نقاط تحتاج مراجعة: [قائمة]
🔴 أخطاء محاسبية: [إن وجدت]
💡 اقتراحات للتحسين: [قائمة]
🔗 ربط القديم بالجديد: [ما تم ربطه]
```

### 5. قواعد محاسبية ثابتة لا تُكسر أبداً
هذه القواعد يجب أن تبقى سارية مهما كانت التعديلات:

1. **لا فاتورة بدون حركة مخزنية مكتملة** (إلا بإذن صريح من المستخدم)
2. **كل قيد يومي متوازن** (مدين = دائن) — لا استثناءات
3. **AVCO يتحدث فقط من فواتير المشتريات المرحّلة** — لا من أوامر الشراء
4. **الفاتورة الجزئية** تعتمد دائماً على (المستلم/المسلّم - المفوتر) — لا على الكمية المطلوبة
5. **الدفع يُسجل عبر نافذة منبثقة** تتيح تحديد المبلغ — لا دفع مباشر بنسبة 100%
6. **الحفظ التلقائي للأوامر الجديدة** يحفظ في المتصفح فقط — لا يُنشئ سجلاً في DB
7. **إشعار الإرجاع (Credit Note)** لا يحرك المخزون — المرتجع المخزني منفصل

## كيف يعمل المراجع عملياً؟
- عند تعديل أي Server Action مالي: أفحص أن القيود لا تزال متوازنة
- عند تعديل مكون UI يخص الفواتير: أتأكد أن الـ props الصحيحة تُمرر
- عند إضافة ميزة جديدة: أربطها بالدورة المستندية الصحيحة وأوضح للمستخدم أين تقع
- في نهاية كل مجموعة تعديلات: أقدم تقرير المراجعة المختصر
